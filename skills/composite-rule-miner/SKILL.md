# 信贷风控复合拦截规则挖掘系统

> **版本**: V1.0 | **创建**: 2026-04-22
> **来源项目**: MTN首贷风险模型（三轮筛选+两轮质疑审查）

## 输入/输出

### 输入（用户提供）

| # | 文件 | 格式 | 说明 |
|---|------|------|------|
| 1 | 规则挖掘结果 | Excel (.xlsx) | 含 `rule_logic` 列，通常由决策树/RF/LightGBM叶子节点提取 |
| 2 | 训练数据 | CSV | train集，含目标变量 |
| 3 | OOT数据 | CSV | 样本外验证集，含目标变量 |
| 4 | 数据字典（可选） | Excel/CSV | 含 feature_name 和中文含义的映射表。如果不提供，通过网络查询特征含义 |
| 5 | 参数（可选） | 对话指定 | 覆盖率范围（默认0.10-0.20）、规则数量（默认2-3条）、目标变量列名（默认'y'） |

### 输出（交付给用户）

```
1. 标准代码块（直接可复制使用）
   ├── query_risk_1 = '...'
   ├── query_risk_2 = '...'  
   ├── query_risk_3 = '...'
   └── query_bad = f"({query_risk_1}) | ({query_risk_2}) | ({query_risk_3})"

2. 验证表格（Train vs OOT 对比）
   ├── Coverage / Precision / Recall / Lift
   └── 差异分析（稳定性证据）

3. 每条规则的业务解释
   ├── 一句话说清楚"为什么这类人风险高"
   └── 三种规则的风险画像差异说明

4. 重叠分析
   ├── 每条规则独有命中人数
   └── 规则间重叠率
```

## 工作流程

```
输入文件 → Phase 1(理解) → Phase 2(特征理解) → Phase 3(验证筛选) → Phase 4(组合搜索) → Phase 5(业务审查) → Phase 6(输出)
```

### Phase 1: 理解需求

确认以下信息：
1. 三个文件路径（规则Excel、train CSV、oot CSV）
2. 目标变量列名（默认 `y`）
3. 覆盖率范围（默认 0.10-0.20）
4. 规则数量（默认 2-3 条）
5. 是否有数据字典

### Phase 2: 理解特征含义（关键步骤）

**这一步是业务可解释性的基础。**

```
用户提供了数据字典？
    │
    ├── 是 → 加载数据字典，建立 feature_name → 业务含义 的映射
    │
    └── 否 → 对规则中出现的关键特征，通过网络搜索理解其业务含义
             例如: "mom_bill_payment_amt_3m" → 搜索 "mobile money bill payment amount"
             或搜索该特征所属系统的官方文档
```

**目标**：对每条候选规则中的每个特征，都能用业务语言解释其含义。

### Phase 3: 筛选与验证

#### 3.1 筛选硬性约束

| 约束项 | 标准值 | 原因 |
|--------|--------|------|
| 单条规则覆盖率 | >= 3% | 太低则OOT不稳定 |
| 单条规则 OOT lift | >= 1.05 | OOT上仍有效果 |
| precision 差异(train vs OOT) | < 7% | 稳定性底线 |
| coverage 差异(train vs OOT) | < 6% | 稳定性底线 |

#### 3.2 业务可解释性过滤（核心价值）

对每条通过硬性约束的规则，逐条检查：

**通过条件**：能用一句话说清楚"满足这些条件的人为什么风险更高"

**淘汰信号**：
- 特征之间无业务关联（纯数据拟合）
- 阈值过于精确像 grid search 出来的（如 748.50、63.50）
- 条件方向与业务常识矛盾（如"P2P转账多=高风险"但数据证明P2P越多坏账率越低）

**按风险画像分类**（确保规则间低重叠）：

| 画像类型 | 判断方法 | 设计方向 |
|---------|---------|---------|
| 行为恶化型 | 曾经正常但近期骤降 | 比较不同时间窗口的同一指标 |
| 有能力无意愿型 | 有收入/存款但不还款 | 收入指标高 + 还款指标低 |
| 网络孤立型 | 交易对象极少 | uniq_acceptors 类指标低 |
| 资金流出型 | 只取不存/只花不赚 | withdrawal高 + deposit低 |
| 长期缺失型 | 某类金融行为长期为零 | 关键行为指标 = 0 或极低 |

#### 3.3 特征方向性验证（防止方向性错误）

对规则中出现的关键特征，**必须**分段验证其与坏账率的关系：

```python
# 例如验证 mom_bill_payment_amt_3m
for low, high in [(0,100), (100,300), (300,1000), (1000,999999)]:
    subset = df.query(f'feature > {low} and feature <= {high}')
    print(f"{low}-{high}: 坏账率={subset['y'].mean():.2%}")
```

**确认规则中的条件方向与数据实际方向一致。** 如果规则说"X>50=高风险"但数据证明X越大坏账率越低，则该规则必须淘汰。

### Phase 4: 组合搜索

从所有通过业务审查的稳定规则中，搜索最优组合：

1. `itertools.combinations` 搜索 C(N, K) 种组合
2. 对每个组合用 OR（`|`）连接，在 train + OOT 上验证
3. 过滤：覆盖率在目标范围内 + lift达标 + 稳定性通过
4. 计算重叠率：`1 - 并集人数 / 各规则命中人数之和`
5. 按综合得分排序：`score = oot_recall - 2 * stability`

**搜索代码参考** `scripts/rule_search.py`

### Phase 5: 业务审查（质疑环节）

对最终推荐的组合方案，从以下维度做压力测试：

| 维度 | 检查内容 |
|------|---------|
| 因果逻辑 | 特征组合是真有因果还是碰巧相关？能否举反例？ |
| 阈值合理性 | 阈值有业务依据吗？±10%变化时效果是否平滑？ |
| 规则间一致性 | 多条规则间有无逻辑矛盾？描述的是否不同人群？ |
| 监管可解释 | 向监管能否一句话说清楚每条规则的逻辑？ |
| 覆盖率合理性 | 组合覆盖率是否在目标范围内？去掉最弱规则后效果？ |

### Phase 6: 输出

按「输出」章节的格式交付。

## 跨项目复用指南

本技能不绑定任何特定行业或数据源。适配新项目时：

| 步骤 | 做什么 |
|------|--------|
| 理解特征 | 通过数据字典或网络搜索，理解每个特征的业务含义 |
| 识别关键维度 | 确定数据中有哪些行为维度（如金融行为、通信行为、消费行为等） |
| 确定风险画像 | 根据业务场景设计风险画像分类（参考Phase 3的分类思路，但不限于此） |
| 验证方向性 | 对关键特征做分段坏账率分析，确认方向正确 |

**示例**：
- 通信+移动支付项目 → 五种画像如上所述
- 电商信贷项目 → 画像可能是：消费降级型、退货异常型、多平台借贷型、虚假交易型
- 银行信用卡项目 → 画像可能是：额度使用率突变型、最低还款型、套现嫌疑型、沉默激活型

## 关键经验教训

### 从MTN项目学到的（通用原则）

1. **业务解释是铁底线**：不能解释的规则，数据再好也不能上
2. **纯统计规则OOT不稳定**：没有业务逻辑支撑的规则，跨时间泛化能力差
3. **特征方向性必须验证**：不能假设"某个特征越大/越小=风险越高"，必须用数据验证
4. **规则间互补 > 单条强度**：3条各自覆盖5%且不重叠，比1条覆盖15%更好
5. **覆盖率是组合的结果**：每条规则单独可以很窄（3-8%），OR后达到10-20%
6. **阈值不宜过精确**：如果阈值从700变到800效果差不多，就用整数阈值

### 组合设计核心原则

- 每条规则覆盖**不同风险画像**
- 重叠率 < 15% 优秀，< 25% 可接受
- 每条规则单独覆盖率 3-10% 最合适
- 优先选择特征维度不同的规则组合
- 三条规则的逻辑最好能对应"突然变差型"+"有能力不还型"+"一直很差型"
