## 概述

Chris Risk Workbench 是一个多 Agent 协作的信贷风控工作台。它不是单一模型做所有事情，而是将风控工作拆解为 9 个专业角色，每个角色各司其职、协同作战。

### 核心理念

| 原则 | 说明 |
|------|------|
| **分工明确** | 9 个 Agent 各有专长，不越界 |
| **辩论制衡** | 关键决策触发质疑员 + 跨界顾问辩论，避免盲区 |
| **框架驱动** | 24 个分析方法论知识库，提供方法论支撑 |
| **模式自适应** | 三种工作模式自动识别，也可手动切换 |
| **可定制** | 每个Agent的模型、温度、工具都可配置覆盖 |

### 一句话总结

> Chris 是主编，他只负责对话和调度。分析师看数据，工程师写代码，质疑员挑毛病，研究员查资料，跨界顾问出奇招，框架师推荐方法论，进化师做复盘，视觉员看图片。

## 9 个 Agent 详解

### 1. Chris（主编）

| 属性 | 值 |
|------|-----|
| 角色 | Primary（主 Agent） |
| 模型 | glm-5.1 |
| 温度 | 默认 |
| 颜色 | 🟠 #FF6B35 |

#### 职责

Chris 是你和团队之间的桥梁。他**只对话、只调度、不干活**。

1. **理解需求**：与你对话，理解你的信贷风控需求
2. **分解任务**：将复杂需求拆解为子任务
3. **指派执行**：通过 task 工具派给合适的 Agent
4. **综合结果**：收集各 Agent 的产出，综合成最终答案
5. **审查协调**：用户要求辩论时派质疑员，协调审查-修改-复审流程

#### 调度规则

| 用户需求类型 | Chris 会派给 |
|------------|-------------|
| 数据分析、数据探索 | 分析师 |
| 特征衍生、变量设计 | 工程师（建议两步：设计方案→辩论→代码） |
| 策略制定、规则设计 | 分析师 + 工程师 → 建议辩论审查 |
| 调研、行业研究、竞品分析 | 研究员（串行：先搜后研） |
| 横纵分析、深度研究 | 研究员（parallel-deep-research → hv-analysis） |
| 需要创意突破 | 跨界顾问 |
| 方法论指导 | 框架师 |
| 看图片、截图、证件 | 视觉员 |
| 系统改进、复盘 | 进化师 |
| 整理、同步知识、/neat | 进化师（项目知识 + 跨项目经验沉淀） |
| "辩论一下"/"审查一下" | 质疑员（闸门审查） |

#### 可用工具

- ✅ task（调度子 Agent）
- ✅ skill_mcp, skill
- ✅ read, write
- ❌ grep, bash, look_at（这些活派给别人）

#### 什么时候你会用到 Chris

**永远。** Chris 是你的唯一入口。你不需要直接和其他 Agent 对话，只需告诉 Chris 你想做什么。

### 2. 分析师

| 属性 | 值 |
|------|-----|
| 角色 | subagent |
| 模型 | glm-5.1 |
| 温度 | 0.3（偏精确） |
| 颜色 | 🟢 #4ECDC4 |

#### 核心能力

1. **数据探索**：数据概览、分布分析、缺失值检测、异常值识别
2. **统计分析**：相关性分析、显著性检验、分布检验、假设验证
3. **特征评估**：IV/WOE 计算、PSI 计算、变量稳定性分析、群体一致性检查
4. **数据可视化**：分布图、趋势图、相关性热力图

#### 工作原则

- 先理解数据，再做分析，最后给结论
- 每个结论必须基于数据证据
- 异常发现标注风险等级（高/中/低）
- 输出结构化分析报告：方法 → 发现 → 结论 → 建议

#### 三种模式下的重点

| 模式 | 工作重点 |
|------|---------|
| 数据分析 | 全面探索数据，输出数据质量报告 |
| 特征衍生 | 分析变量的预测能力和稳定性 |
| 策略制定 | 分析策略执行效果数据 |

#### 什么时候 Chris 会派分析师

- "帮我看看这批数据的质量"
- "分析一下这几个变量的分布"
- "计算 IV 值和 WOE"
- "这份策略的执行效果怎么样"

### 3. 质疑员

| 属性 | 值 |
|------|-----|
| 角色 | subagent |
| 模型 | MiniMax-M2.7-highspeed |
| 温度 | 0.5（平衡质疑与理性） |
| 颜色 | 🔴 #E74C3C |
| Skills | critical-thinking-logical-reasoning, debate-review |

#### 核心能力

1. **闸门审查**：在方案设计后、代码实现前拦截问题
2. **审查特征方案**：检查时间穿越、稳定性、覆盖率、冗余、业务含义
3. **审查策略方案**：检查覆盖率、误杀率、冲突、绕过风险、合规性
4. **实证验证**：可写 5-10 行轻量脚本支撑质疑

#### 审查协议

质疑员遵循 `debate-review` Skill 中定义的审查清单和输出格式，使用 🔴🟡🟢✅ 分级标记。

#### 输出格式（结构化审查报告）

1. 🔴 高风险（必须修改）
2. 🟡 中风险（建议修改）
3. 🟢 低风险（可选优化）
4. ✅ 通过审查
5. 总结统计

#### 什么时候 Chris 会派质疑员

- 用户说"辩论一下"/"审查一下"/"帮我看看这个方案"
- 工程师完成特征设计方案后（建议触发）
- 策略规则制定完成后（建议触发）

### 4. 工程师

| 属性 | 值 |
|------|-----|
| 角色 | subagent |
| 模型 | glm-5.1 |
| 温度 | 默认 |
| 颜色 | 🔵 #2196F3 |

#### 核心能力

1. **特征衍生**：根据分析师的结论，设计衍生特征
2. **代码生成**：编写 Python/Pandas 特征工程代码
3. **特征验证**：检查特征的分布、缺失率、相关性
4. **文档输出**：生成特征说明文档

#### 特征衍生方法论

**时间维度：**
- 时间窗口聚合：近1/3/6/12个月的行为统计
- 趋势特征：环比、同比、变化率
- 衰减加权：近期行为权重更高

**交叉维度：**
- 变量组合：A/B、A*B、A-B、A/(A+B)
- 比率衍生：使用频次/授信额度、逾期金额/总金额

**数学变换：**
- 标准化：Z-score、Min-Max 归一化
- 离散化：等频分箱、等宽分箱、决策树分箱
- 对数变换：log(1+x) 处理右偏分布

#### 输出格式（每个特征必须包含）

```
变量名: xxx_3m_overdue_cnt
业务含义: 近3个月逾期次数
计算逻辑: df['overdue_date'].rolling('3M').count()
数据类型: int
预期分布: 右偏，大量0值
```

#### 什么时候 Chris 会派工程师

- "帮我设计逾期相关的衍生特征"
- "根据分析师的结论，衍生一些变量"
- "把这些特征写成 Python 代码"
- "做一下分箱处理"

### 5. 研究员

| 属性 | 值 |
|------|-----|
| 角色 | subagent |
| 模型 | glm-5.1 |
| 温度 | 默认 |
| 颜色 | 🟣 #9B59B6 |
| Skills | parallel-deep-research, hv-analysis, browser-use, agent-browser |

#### 核心能力

1. **深度搜索**：多渠道搜索行业资料、学术论文、监管文件
2. **知识综合**：整理多方信息，提炼关键洞察
3. **趋势分析**：识别行业趋势和新兴风险
4. **竞品分析**：分析同业风控策略和产品
5. **横纵分析**：纵向时间线 + 横向竞品对比的深度研究

#### 🔍 串行研究工作流

研究员的两个核心 Skill 不是二选一，而是**上下游串行协作**：

```
第一步: parallel-deep-research（素材采集）
  → 多源深度搜索，采集高质量素材包
  → 默认使用 ultra-fast 模式（1-10分钟）

第二步: hv-analysis（横纵分析法）
  → 纵轴：沿时间线还原研究对象的完整演变
  → 横轴：在当下截面与竞品系统对比
  → 输出万字级结构化研究报告
```

#### 搜索渠道

- 监管文件：银保监会、央行、地方金融监管局
- 行业报告：咨询公司、行业协会
- 学术论文：风控、机器学习、信用评分
- 新闻动态：财经新闻、风险事件

#### 输出格式

1. 研究主题
2. 信息来源（至少3个）
3. 关键发现（分条列出）
4. 风控启示（对当前项目的具体建议）
5. 不确定性说明

#### 什么时候 Chris 会派研究员

- "调研一下最新的消费金融风控趋势"（串行：先搜后研）
- "分析一下同业的反欺诈策略"（串行：先搜后研）
- "帮我搞懂XX是怎么回事"（hv-analysis 横纵分析）
- "查一下这个监管文件的具体要求"（只用 parallel-deep-research）
- "帮我把这些素材整理成报告"（只用 hv-analysis）

## 辩论审查机制

辩论审查是 Chris Risk Workbench 的核心质量闸门。它不是事后验尸，而是在**方案设计完成后、代码实现前**拦截问题。

### 用户主动触发

辩论由用户主动触发——你说"辩论一下"或"审查一下这个方案"，Chris 就派质疑员进行闸门审查。

### 审查流程

```
工程师/策略师完成方案设计
         │
         ▼
用户说"辩论一下"
         │
         ▼
Chris 派质疑员（传入方案文档 + 数据上下文）
         │
         ▼
质疑员按审查清单逐项检查
  - 特征方案：时间穿越、稳定性、覆盖率、冗余、业务含义...
  - 策略方案：覆盖率、误杀率、绕过风险、合规性...
  - 可写 5-10 行验证脚本做实证支撑
         │
         ▼
输出结构化审查报告（🔴高风险 🟡中风险 🟢低风险 ✅通过）
         │
         ▼
Chris 根据报告决定下一步：
  🔴 → 派原作者修改 → 再派质疑员复审
  🟡 → 问用户是否修改
  ✅ → 通过，进入代码实现
```

### 审查协议

质疑员的审查协议定义在 `skills/debate-review/SKILL.md` 中，包含：
- 前置检查（方案完整性验证）
- 特征审查清单（7 项）
- 策略审查清单（9 项）
- 结构化输出格式（编号追踪，便于复审）
- 复审机制（只审之前的问题项）

修改审查清单只需编辑 SKILL.md，无需改代码。

## 三种工作模式

### 自动检测

插件启动时会扫描项目目录结构，自动判断工作模式：

| 目录结构 | 检测到的模式 |
|---------|------------|
| 有 `data/` + `features/` | 特征衍生模式 |
| 有 `data/` + `strategy/` | 策略制定模式 |
| 只有 `data/` | 数据分析模式 |
| 其他 | 综合模式（all） |

### 模式说明

#### 数据分析模式（data-analysis）

**场景**：你有一批新数据，需要全面了解数据质量、分布特征、异常值。

**Chris 的调度优先级**：
1. 分析师 → 探索数据
2. 框架师 → 推荐分析方法论
3. 质疑员 → 审查分析结论

#### 特征衍生模式（feature-engineering）

**场景**：你需要基于现有数据设计衍生特征，生成特征工程代码。

**Chris 的调度优先级**：
1. 分析师 → 评估变量预测能力
2. 工程师 → 设计衍生特征 + 生成代码
3. 质疑员 + 跨界顾问 → 辩论特征合理性
4. 进化师 → 评估特征质量

#### 策略制定模式（strategy-design）

**场景**：你需要制定信贷风控策略（准入、额度、定价、催收等）。

**Chris 的调度优先级**：
1. 分析师 → 分析策略执行效果数据
2. 研究员 → 搜索行业对标策略
3. 工程师 → 实现策略逻辑代码
4. 质疑员 + 跨界顾问 → 辩论策略合理性
5. 进化师 → 评估策略效果

### 案例二：特征衍生 — 逾期行为特征设计

**背景**：你有一份信贷客户的还款行为数据，需要设计逾期相关的衍生特征用于信用评分模型。

**对话过程**：

```
👤 用户：我有一份客户的还款行为数据，需要设计逾期相关的衍生特征，用于A卡评分模型

🤖 Chris：这是一个特征衍生任务。我的计划：
         1. 派框架师推荐适合的特征工程方法论
         2. 派分析师评估现有变量的预测能力
         3. 派工程师设计衍生特征并生成 Python 代码
         4. 触发辩论：质疑员 + 跨界顾问审查特征设计
         5. 进化师做最终评估

         先让框架师推荐方法论...

🤖 框架师：
         推荐以下3个框架（按优先级）：

         1. **RFM模型**（优先级：高）
            - R(Recency)：最近一次逾期距今天数
            - F(Frequency)：历史逾期次数
            - M(Monetary)：历史逾期金额
            - 适合对行为数据做结构化特征衍生

         2. **时间窗口聚合**（优先级：高）
            - 近1/3/6/12个月的逾期统计
            - 捕捉行为的时间衰减特性

         3. **评分卡策略**（优先级：中）
            - WOE分箱需要考虑特征的稳定性
            - IV值筛选时注意Y的定义一致性

🤖 分析师：（分析现有变量后）
         基础变量的预测能力评估：
         - 逾期天数：IV = 0.45（强预测）
         - 逾期金额：IV = 0.32（中等预测）
         - 还款日偏移：IV = 0.28（中等预测）
         ⚠️ 历史最大逾期天数 PSI = 0.15，稳定性一般

         建议衍生方向：时间窗口聚合、衰减加权、比率衍生

🤖 工程师：（生成代码）
         ## 衍生特征清单

         ```python
         import pandas as pd
         import numpy as np

         def derive_overdue_features(df):
             features = pd.DataFrame(index=df.index)

             # === 时间窗口聚合 ===
             for window in ['1M', '3M', '6M', '12M']:
                 features[f'od_cnt_{window}'] = (
                     df['overdue_flag']
                     .rolling(window, on=df['overdue_date'])
                     .sum()
                 )
                 features[f'od_amt_max_{window}'] = (
                     df['overdue_amount']
                     .rolling(window, on=df['overdue_date'])
                     .max()
                 )

             # === 比率衍生 ===
             features['od_amt_ratio'] = (
                 df['overdue_amount'] / (df['total_amount'] + 1)
             )
             features['od_cnt_to_credit'] = (
                 features['od_cnt_12M'] / (df['credit_lines'] + 1)
             )

             # === 趋势特征 ===
             features['od_trend_3m_12m'] = (
                 features['od_cnt_3M'] / (features['od_cnt_12M'] + 1)
             )

             return features
         ```

         | 变量名 | 业务含义 | 预期分布 |
         |--------|---------|---------|
         | od_cnt_3M | 近3个月逾期次数 | 右偏，大量0值 |
         | od_amt_max_6M | 近6个月最大逾期金额 | 右偏长尾 |
         | od_amt_ratio | 逾期金额占比 | 集中在0附近 |
         | od_trend_3m_12m | 短期/长期逾期趋势 | 接近0-1区间 |

🤖 Chris：特征设计完成，现在触发辩论。

🤖 质疑员：
         1. **质疑点**：od_trend_3m_12m 在 12 个月无逾期时会变成 0/1=0，
            无法区分"一直良好"和"最近好转"
         2. **风险**：这个特征可能低估了"从差变好"的客户的真实风险改善
         3. **建议**：增加一个分段变量，区分趋势方向（改善/恶化/稳定）

🤖 跨界顾问：
         从保险行业的精算模型借鉴——他们用"索赔频率-严重度"分离建模：
         - 频率模型：逾期是否发生（二分类）
         - 严重度模型：发生逾期时的金额（条件回归）
         这种分离可以避免零膨胀问题。

🤖 Chris：综合辩论结果，最终方案：
         1. 保留工程师的原始特征设计
         2. 新增质疑员建议的趋势方向分段变量
         3. 参考跨界顾问的建议，将频率和严重度分开建模
         4. 工程师请据此更新代码

🤖 工程师：（更新代码，新增特征）
         已补充：
         - od_trend_direction: 趋势方向（improving/stable/worsening）
         - od_severity_avg: 条件平均逾期金额（仅逾期时段）
```

**涉及 Agent**：Chris → 框架师 → 分析师 → 工程师 → 质疑员 → 跨界顾问 → Chris 综合 → 工程师更新

🤖 工程师：
         ## 准入策略优化方案

         def optimized_admission(customer):
             # 基础规则（不变）
             if customer.age < 22 or customer.age > 55:
                 return reject("年龄不符")

             # 优化：硬查询区分类型
             loan_inquiries = customer.inquiries_3m(type="loan")
             card_inquiries = customer.inquiries_3m(type="credit_card")

             if loan_inquiries > 8:  # 原来>6，放开了贷款查询
                 return reject("贷款查询过多")
             if card_inquiries > 4:  # 新增信用卡查询上限
                 return reject("信用卡查询过多")

             # 新增：灰名单机制
             if customer.credit_score < 500:
                 return reject("综合评分不足")
             elif customer.credit_score < 580:
                 return conditionally_approve(
                     reduced_limit=True,
                     enhanced_monitoring=True
                 )  # 灰名单：通过但降低额度和加强监控

             return approve()
         ```

## 常见问题

### Q：我需要直接和某个 Agent 对话吗？

不需要。所有对话都通过 Chris 进行。Chris 会根据你的需求自动调度合适的 Agent。你只需要用自然语言描述你想做什么。

### Q：辩论机制会拖慢速度吗？

辩论只在关键决策点触发，不是每次对话都会辩论。典型的触发场景是：
- 分析师给出结论后
- 工程师完成特征设计后
- 策略制定完成后

辩论轮次默认最多 3 轮，可通过配置调整。

### Q：如何禁用某个 Agent？

在配置文件中设置：
```jsonc
{
  "disabled_agents": ["进化师"]
}
```

### Q：如何更换 Agent 的模型？

```jsonc
{
  "agents": {
    "分析师": {
      "model": "zai-coding-plan/glm-5.1",
      "temperature": 0.2
    }
  }
}
```

### Q：如何添加自定义框架？

在 `frameworks/` 目录下新增 `.txt` 文件即可。文件名就是框架名，文件内容是框架说明。

例如创建 `frameworks/L2-business/自定义框架.txt`：
```
自定义框架名称

## 框架说明
...

## 使用步骤
...
```

插件启动时会自动加载。

### Q：三种工作模式有什么区别？

| 模式 | 侧重点 | 主要调度 Agent |
|------|--------|--------------|
| 数据分析 | 数据探索和统计 | 分析师、框架师 |
| 特征衍生 | 变量设计和代码 | 分析师、工程师、质疑员 |
| 策略制定 | 规则设计和优化 | 研究员、分析师、工程师 |
| 综合 | 全部能力可用 | 按需调度 |

### Q：插件安装后如何验证是否正常？

在 OpenCode 中输入任何风控相关问题，如果看到日志中出现 `[chris-risk-workbench] === chris-risk-workbench ready ===`，说明插件加载成功。
